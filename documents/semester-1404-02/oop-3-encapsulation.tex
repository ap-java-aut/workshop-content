\chapter{\DocTitle}
\section{مقدمه}

کپسوله‌سازی \lr{(Encapsulation)} یکی دیگر از اصول بنیادین شی‌گرایی است.
این مفهوم امکان پنهان‌سازی جزئیات پیاده‌سازی، شامل فیلدها، متدها و کلاس‌ها
را از دید سایر کلاس‌ها فراهم می‌سازد. به بیان دیگر، کپسوله‌سازی داده و کد
مربوط به آن را به‌عنوان یک واحد منسجم در نظر می‌گیرد که همان کلاس تعریف‌شده
است و یکی از ویژگی‌های آن، پنهان‌سازی جزئیات پیاده‌سازی می‌باشد.

در این بخش، با اصل کپسوله‌سازی و متدهای \lr{\texttt{getter}} و
\lr{\texttt{setter}} به‌طور کامل آشنا خواهید شد.

\section{چرا از \lr{Encapsulation} استفاده می‌کنیم؟}

برای روشن‌تر شدن موضوع، با یک مثال آغاز می‌کنیم. در تمرین کارگاه 5، پیش‌تر
برای تعریف فیلدها و متدها از کلیدواژه \lr{\texttt{public}} استفاده
می‌کردید. برای نمونه، در کلاس \lr{Person} فیلدها به صورت زیر تعریف
می‌شدند:

\begin{codebox}{java}
public class Person {
    public String name;
    public String nationalID;
    public String phoneNumber;
    public static int totalPersons;
    
}
\end{codebox}

این شیوه بدین معناست که در کلاسی دیگر، برای مثال \lr{\texttt{Main}} ،
می‌توان با ساختن نمونه‌ای از کلاس \lr{\texttt{Person}} و تنها با استفاده
از عملگر نقطه به فیلد \lr{\texttt{nationalID}} دسترسی داشت، یعنی اگر
آبجکتی از جنس \lr{\texttt{Person}} به نام \lr{\texttt{ali}} داشتیم
کافیست با \lr{\texttt{ali.nationalID}} به فیلد کد ملی \lr{\texttt{ali}}
دسترسی پیدا کرد. دقیقاً همین وضعیت برای متدها نیز برقرار است. اما این
گونه تعریف فیلدها اشکالاتی به همراه دارد. برای نمونه، فرض کنید در
\lr{\texttt{Main}} شخصی با یک کد ملی مشخص تعریف نموده‌اید که یکتا است و
همچنین آن شخص دارای شماره تلفنی نیز هست. اکنون به‌سادگی می‌توانید کد ملی
را مستقیما داخل \lr{\texttt{Main}} تغییر دهید، در نتیجه شماره تلفن که
برای فرد دیگری بوده حال برای کسی با کد ملی متفاوتی تعلق می‌گیرد یا حتی یک
کد ملی نامعتبر برای آن درج می‌شود. بنابراین، دسترسی آزاد به این فیلدها و
متدهای \lr{\texttt{public}} در کلاس‌های دیگر، امکان تغییر آسان مقادیر
آن‌ها را فراهم می‌سازد که میتواند موجب به آلوده شدن دیتا شما (کد ملی
نامعتبر) یا حتی نادرست شدن دیتا شما شود (کد ملی شخصی دیگر برای یک شماره
تلفن).

برای همین اصل کپسوله‌سازی مطرح می‌شود. این اصل دارای خاصیتی است که وضعیت
درونی اشیاء، مانند فیلدهایشان، را پنهان می‌کند. به این ویژگی، اصطلاحاً
پنهان‌سازی داده \lr{(Data Hiding)} گفته می‌شود. در ادامه، با بررسی کد، این
مفاهیم ملموس‌تر خواهند شد.

\section{چطور در کد از \lr{Encapsulation} استفاده کنیم؟}

تاکنون با ساختار کلی یک کلاس و روش تعریف و ساخت آن آشنا شده‌اید. اکنون
چگونگی اعمال پنهان‌سازی داده را بررسی می‌کنیم.

\subsection{کلیدواژه \texttt{private}}

فرض کنید قصد شبیه‌سازی یک باغ وحش را داریم. کلاسی به نام
\lr{\texttt{Zoo}} تعریف می‌کنیم و فیلدهای زیر را در آن قرار می‌دهیم (توصیه
می‌شود تمام کدها را خود نیز اجرا کنید):

\begin{codebox}{java}
public class Zoo {
    private String name;
    private ArrayList<Animal> animals;
    
    public Zoo(){  
    }  
}
\end{codebox}

اکنون کلاس \lr{\texttt{Main}} را تعریف کرده و نمونه‌ای از کلاس
\lr{\texttt{Zoo}} می‌سازیم. تلاش می‌کنیم فیلد \lr{\texttt{name}} را
مقداردهی کنیم:

\begin{codebox}{java}
public class Main {  
    public static void main(String[] args) {  
        Zoo zoo = new Zoo();  
        zoo.name = "Woodland Park Zoo" ;  
    }  
}
\end{codebox}

با خطا مواجه می‌شوید. جالب‌تر آنکه حتی پس از درج نقطه در کنار نام شی
\lr{\texttt{zoo}} ، \lr{IDE} این فیلد را در فهرست پیشنهادی خود نمایش
نمی‌دهد.

در رابطه با وراثت، وقتی کلاس والد یک فیلد \lr{\texttt{private}} دارد،
وضعیت چگونه است؟

تفاوتی ندارد؛ فیلدهای \lr{\texttt{private}} حتی در کلاس‌های فرزند نیز
به‌صورت مستقیم قابل دسترسی نیستند و نمی‌توان آن‌ها را فراخوانی نمود. متدهای
\lr{\texttt{private}} نیز دقیقاً همین وضعیت را دارند. این متدها و فیلدها
تنها در درون کلاس خود قابل دسترسی هستند اما در کلاس فرزند غیرقابل
دسترسی.

گروه خاصی از متدها، سازنده‌ها \lr{(Constructors)} هستند. آیا می‌توان
کانستراکتورها را نیز \lr{\texttt{private}} تعریف کرد؟

پاسخ مثبت است؛ به‌ویژه زمانی که بخواهیم ساخت نمونه از کلاس را محدود
نماییم. بدیهی است که با چنین کانستراکتوری نمی‌توان نمونه‌ای ساخت. یک مثال
عالی، کلاس \lr{\texttt{Math}} در جاوا است که کانستراکتوری
\lr{\texttt{private}} دارد و تمامی متدهای آن \lr{\texttt{static}} هستند.
می‌توانید در \lr{IntelliJ} این کلاس را \lr{import} کرده، بر روی
\lr{\texttt{Math}} راست‌کلیک نموده و از طریق \lr{go to} ، گزینه
\lr{Declaration or usages} را انتخاب کنید تا به سورس کلاس دست یابید:

\begin{figure}[H]
\centering
\includegraphics[width=0.8\textwidth]{images/oop-3-2.jpg}
\end{figure}

نکته جالب دیگر، \lr{\texttt{final class}} است. وقتی این کلیدواژه برای یک
کلاس به کار می‌رود، بدین معناست که آن کلاس دیگر قابلیت ارث‌بری ندارد.

اگر زمانی نیاز به ساخت نمونه از کلاسی با کانستراکتور
\lr{\texttt{private}} داشته باشید، اولاً باید حتماً در داخل همان کلاس
نمونه را بسازید؛ ثانیاً یا باید این کار را از طریق متدی دیگر انجام دهید
یا از تابع \lr{\texttt{getInstance()}} استفاده کنید. تابع
\lr{\texttt{getInstance()}} روشی برای پیاده‌سازی الگوی \lr{Singleton}
است. در این الگو، کلاس تنها یک نمونه از خود می‌سازد و دسترسی به آن را از
طریق همین تابع فراهم می‌کند. \lr{\texttt{getInstance()}} بررسی می‌کند که
آیا نمونه‌ای از کلاس از پیش ساخته شده یا خیر؛ اگر خیر، یک نمونه جدید
ایجاد و ذخیره می‌کند و سپس آن را بازمی‌گرداند. بدین ترتیب همواره تنها یک
شی از آن کلاس در سراسر برنامه وجود خواهد داشت. توجه شد که منظور از الگوی
\lr{Singleton} این است که این الگو اطمینان می‌دهد که از یک کلاس تنها یک
نمونه در سراسر برنامه وجود داشته باشد و دسترسی سراسری به آن فراهم گردد
دقیقا کاری که تابع \lr{\texttt{getInstance()}} محقق میکند.

حال این پرسش مطرح می‌شود که آیا کلاس‌ها نیز می‌توانند \lr{\texttt{private}}
باشند؟

به‌طور کلی، بله. اما نه هر کلاسی. تنها زمانی که کلاس‌های تودرتو
\lr{(nested)} تعریف می‌کنید، \lr{inner class} ها می‌توانند
\lr{\texttt{private}} باشند و دسترسی به آن‌ها صرفاً از طریق
\lr{outer class} ممکن است. همچنین امکان ارث‌بری از این کلاس‌ها وجود ندارد
(مانند \lr{final class} ها). افزون بر این، \lr{outer class} ها نمی‌توانند
\lr{\texttt{private}} باشند. در مواردی که برای مثال سه کلاس تودرتو دارید
(بیرونی‌ترین آن‌ها \lr{outer class} است)، می‌توانید هر دو یا یکی از
\lr{inner class} ها را \lr{\texttt{private}} نمایید. تعریف کلاس معمولی
به‌صورت \lr{\texttt{private}} بی‌معناست، زیرا عملاً استفاده از آن کلاس را
غیرممکن می‌سازد. توجه کنید که جاوا به ما اجازه ساخت کلاس داخل یک کلاس را
میدهد که به این کار \lr{nested class} گفته میشه یا به فارسی کلاس های
تودرتو میگویند. برای درک بهتر محدودیت‌های سطوح دسترسی در کلاس‌های تودرتو
\lr{(nested)} و نحوه نوشتن آن، مثال زیر را در نظر بگیرید:

\begin{codebox}{java}
public class Zoo {                     // outer class, cannot be private
    private String name;
    
    private class AnimalCensus {       // inner class, can be private or public
        private int count;
        void increment() { count++; }
    }
    
    public class Exhibit {             // inner class, can be private or public
        public String exhibitName;
    }
}
\end{codebox}

در این ساختار، \lr{outer class} (\lr{\texttt{Zoo}}) همواره باید
\lr{\texttt{public}} یا \lr{package-private} باشد و نمی‌توان آن را
\lr{\texttt{private}} تعریف کرد. در مقابل، \lr{inner class} ها می‌توانند
\lr{\texttt{private}} (مانند \lr{\texttt{AnimalCensus}}) یا
\lr{\texttt{public}} (مانند \lr{\texttt{Exhibit}}) باشند. کلاس
\lr{\texttt{AnimalCensus}} فقط در درون \lr{\texttt{Zoo}} قابل استفاده
است، درحالی‌که \lr{\texttt{Exhibit}} از بیرون نیز در دسترس خواهد بود.

با انجام این مراحل، شما پنهان‌سازی داده را محقق ساخته‌اید. اما پرسش اساسی
این است که چگونه می‌توان به این متدها یا فیلدهای \lr{\texttt{private}}
دسترسی داشت؟ آیا هر بار باید آن‌ها را مجدداً \lr{\texttt{public}} کنیم و
صورت مسئله را پاک کنیم؟ قطعاً خیر.

برای دسترسی و تغییر فیلدهای \lr{\texttt{private}} از متدهای \lr{getter}
و \lr{setter} استفاده می‌شود.

\subsection{متدهای \lr{getter}}

به‌طور خلاصه، متدهای \lr{getter} مقدار یک فیلد \lr{\texttt{private}} را
بازمی‌گردانند. برای مشخص‌سازی یک متد به‌عنوان \lr{getter} ، نام آن با
\lr{\texttt{get}} آغاز می‌شود. اکنون به کد مربوطه می‌پردازیم.

ابتدا کلاس \lr{\texttt{Animal}} را به‌صورت زیر تعریف کنید و متدهای
\lr{getter} را برای آن قرار دهید:

\begin{codebox}{java}
import java.util.ArrayList;
public class Animal {
    private String name;
    private int age;
    private String species;

    public Animal(String name, int age, String species) {
        this.name = name;
        this.age = age;
        this.species = species;
    }
    public String getName() {
        return name;
    }
    public String getSpecies() {
        return species;
    }
    public int getAge() {
        return age;
    }
    @Override
    public String toString() {
        return "Name: " + name + ", Age: " + age + ", Species: " + species;
    }
}
\end{codebox}

چرا متد \lr{\texttt{toString()}} را \lr{override} می‌کنیم؟ هنگام استفاده
از متد \lr{\texttt{toString()}} کلاس \lr{\texttt{Object}} برای چاپ یک
شی، خروجی نامناسبی تولید می‌شود. برای چاپ اطلاعات خوانا و مورد نیاز، کافی
است این متد را بازنویسی کرده و خروجی مطلوب را برگردانیم. برای مثال در
کلاس \lr{\texttt{Main}} یک شی از این کلاس بسازید و
\lr{\texttt{toString()}} را فراخوانی و چاپ کنید؛ خروجی به شکل زیر خواهد
بود:

\begin{figure}[H]
\centering
\includegraphics[width=0.8\textwidth]{images/oop-3-3.png}
\end{figure}

در ادامه کلاس \lr{\texttt{Panda}} را نیز به‌عنوان فرزند تعریف کرده و
متدهای \lr{getter} را در آن قرار دهید:

\begin{codebox}{java}
public class Panda extends Animal {
    private String whatTheyEat;
    private int numberOfChildren;

    public Panda(String name, int age, String species, String whatTheyEat, int numberOfChildren) {
        super(name, age, species);
        this.whatTheyEat = whatTheyEat;
        this.numberOfChildren = numberOfChildren;
    }
    public int getNumberOfChildren() {
        return numberOfChildren;
    }
    public String getWhatTheyEat() {
        return whatTheyEat;
    }
}
\end{codebox}

اکنون در کلاس \lr{\texttt{Main}} کد زیر را بنویسید و اجرا کنید:

\begin{codebox}{java}
    public static void main(String[] args) {
        Panda panda = new Panda("whitey", 2, "Giant Panda", "Bambo", 0) ;
        System.out.println(panda.getName());
        System.out.println(panda.getAge());
        System.out.println(panda.getSpecies());
        System.out.println(panda.getWhatTheyEat());
        System.out.println(panda.getNumberOfChildren());
    }
\end{codebox}

تمامی اطلاعات چاپ می‌شود.

تمامی متدهایی که در کلاس‌های \lr{\texttt{Animal}} و \lr{\texttt{Panda}}
با \lr{\texttt{get}} شروع می‌شوند، \lr{getter} های ما هستند و همان‌گونه که
انتظار می‌رود، مقادیر فیلدهای شی \lr{\texttt{panda}} را برمی‌گردانند. نکته
ظریف این است که شی \lr{\texttt{panda}} توانست از \lr{getter} های کلاس
والد، برای مثال \lr{\texttt{getName()}} استفاده کند؛ بنابراین
\lr{getter} ها نیز همانند متدهای معمولی به ارث می‌رسند و تنها وظیفه آن‌ها
برگرداندن مقدار فیلدهاست.

\subsection{متدهای \lr{setter}}

وظیفه این متدها مقداردهی و به‌روزرسانی یا تغییر مقدار فیلدهای
\lr{\texttt{private}} است. نکته کلیدی در منطق استفاده از \lr{setter} آن
است که با کاهش سطح دسترسی به فیلد، از تغییر مقدار آن توسط هر بخشی از کد
جلوگیری می‌شود. بنابراین ضروری است که در این متدها شرطی برای بررسی معتبر
بودن مقدار ورودی قرار داده شود. نام این متدها نیز با \lr{\texttt{set}}
آغاز می‌شود.

در ادامه، متدهای \lr{setter} را به کلاس \lr{\texttt{Animal}} اضافه کنید:

\begin{codebox}{java}
public void setSpecies(String species) {
        this.species = species;
}
public void setAge(int age) {
    if(age>=0)
        this.age = age;
    else{
        System.out.println("Age can't be negative");
    }
}
public void setName(String name) {
    if(Character.isUpperCase(name.charAt(0)))
        this.name = name;
    else{
        System.out.println("Name should start with an upper-case letter");
    }
}
\end{codebox}

و کلاس \lr{\texttt{Panda}} را نیز به شکل زیر تغییر داده و متدهای
\lr{setter} را اضافه نمایید:

\begin{codebox}{java}
public void setNumberOfChildren(int numberOfChildren) {
    if(numberOfChildren>=0)
        this.numberOfChildren = numberOfChildren;
    else{
        System.out.println("Number of children can't be negative");
    }
}
public void setWhatTheyEat(String whatTheyEat) {
    this.whatTheyEat = whatTheyEat;
}
\end{codebox}

این قسمت را نیز به کلاس \lr{\texttt{Main}} اضافه کرده و نتیجه را مشاهده
کنید:

\begin{codebox}{java}
        System.out.println("----------------------------");
        panda.setName("cutie");
        panda.setAge(1);
        panda.setSpecies("Red Panda");
        panda.setWhatTheyEat("mushrooms");
        panda.setNumberOfChildren(1);
        System.out.println(panda.getName());
        System.out.println(panda.getAge());
        System.out.println(panda.getSpecies());
        System.out.println(panda.getWhatTheyEat());
        System.out.println(panda.getNumberOfChildren());
\end{codebox}

شاید این پرسش برای شما مطرح شود که با وجود متدهای \lr{getter} و
\lr{setter} ، کماکان امکان تغییر فیلدها وجود دارد؛ بنابراین چه تفاوتی با
حالت \lr{\texttt{public}} بودن آن‌ها دارد؟ تفاوت اساسی در این است که
\lr{getter} و \lr{setter} یک لایه کنترلی بر روی دسترسی ایجاد می‌کنند: در
\lr{setter} می‌توان پیش از مقداردهی، اعتبارسنجی انجام داد یا از تغییر
مقدارهای نامعتبر جلوگیری کرد. همچنین اگر فقط \lr{getter} تعریف شود، فیلد
صرفاً خواندنی \lr{(read-only)} خواهد بود که در صورت نیاز آن را برای افراد
با صلاحیت مشخصی نشان میتوان داد (مانند ادمین های یک سایت). افزون بر این،
پیاده‌سازی درونی کلاس می‌تواند تغییر کند بی‌آنکه کدهای بیرونی متأثر شوند.
در نتیجه، دسترسی به داده‌ها اگرچه میسر است، اما کاملاً تحت نظارت کلاس
خواهد بود. همچنین در بسیاری از موارد، تمامی فیلدهای
\lr{\texttt{private}} نیازی به متدهای \lr{getter} و \lr{setter} ندارند.
برای مثال، فرض کنید در کلاس \lr{\texttt{Zoo}} فیلدی به نام
\lr{\texttt{animalCount}} به‌صورت \lr{\texttt{private}} تعریف شده که صرفاً
برای ردیابی داخلی تعداد حیوانات به‌کار می‌رود:

\begin{codebox}{java}
public class Zoo {
    private String name;
    private ArrayList<Animal> animals;
    private int animalCount;

    public Zoo() {
        animals = new ArrayList<>();
        animalCount = 0;
    }

    public void addAnimal(Animal a) {
        animals.add(a);
        animalCount++;
    }
}
\end{codebox}

در اینجا، تغییر \lr{\texttt{animalCount}} منحصراً از طریق متد
\lr{\texttt{addAnimal}} انجام می‌شود و هیچ کلاس بیرونی نباید مستقیماً آن
را بخواند یا تغییر دهد. ارائه \lr{getter} یا \lr{setter} برای چنین فیلدی
نه‌تنها ضرورتی ندارد، بلکه می‌تواند کپسوله‌سازی را تضعیف کند و امکان
دستکاری ناخواسته در وضعیت درونی شی را فراهم آورد. بنابراین، تنها
فیلدهایی به \lr{getter} و \lr{setter} مجهز می‌شوند که واقعاً نیاز به
دسترسی کنترل‌شده از بیرون داشته باشند. پس در برنامه های جاوا همواره نیازی
به متد های \lr{setter} و \lr{getter} برای تمامی فیلد های پرایوت نیست.

\subsection{کلیدواژه \lr{protected}}

اگر فیلدی را \lr{\texttt{protected}} تعریف کنیم، تنها کلاس‌های فرزند و
کلاس‌های درون همان پکیج به آن دسترسی دارند. سایر کلاس‌ها برای دسترسی باید
از \lr{getter} و برای تغییر از \lr{setter} استفاده کنند. به عبارت دیگر،
سطح دسترسی \lr{\texttt{protected}} میان \lr{\texttt{public}} و
\lr{\texttt{private}} قرار می‌گیرد. همان‌طور که پیش‌تر بیان شد، فیلد یا متد
\lr{\texttt{private}} حتی در کلاس‌های فرزند نیز قابل دسترسی نیست. با
استفاده از \lr{\texttt{protected}} این محدودیت برطرف می‌شود.

اکنون کلاس \lr{\texttt{Animal}} را به صورت زیر تعریف کنید:

\begin{codebox}{java}
import java.util.ArrayList;
public class Animal {
    protected String name;
    protected int age;
    protected String species;

    public Animal(String name, int age, String species) {
        this.name = name;
        this.age = age;
        this.species = species;
    }
    @Override
    public String toString() {
        return "Name: " + name + ", Age: " + age + ", Species: " + species;
    }
}
\end{codebox}

کلاس \lr{\texttt{Panda}} نیز به شکل زیر تغییر می‌یابد:

\begin{codebox}{java}
public class Panda extends Animal {
    protected String whatTheyEat;
    protected int numberOfChildren;
    public Panda(String name, int age, String species, String whatTheyEat, int numberOfChildren) {
        super(name, age, species);
        this.whatTheyEat = whatTheyEat;
        this.numberOfChildren = numberOfChildren;
    }
    public void show(){
        System.out.println(super.name);
        System.out.println(super.age);
        System.out.println(super.species);
        System.out.println(this.whatTheyEat);
        System.out.println(this.numberOfChildren);
    }
}
\end{codebox}

سپس در \lr{\texttt{Main}} کد زیر را قرار دهید:

\begin{codebox}{java}
import java.util.ArrayList;  
import java.lang.*;  
public class Main {    
    public static void main(String[] args) {  
        Panda panda = new Panda("whitey", 2, "Giant Panda", "Bambo", 0) ;  
        panda.show();  
    }  
}
\end{codebox}

همان‌طور که ملاحظه می‌کنید، برنامه بدون مشکل اجرا می‌شود. دلیل آن است که
فیلدهای \lr{\texttt{protected}} در کلاس فرزند قابل دسترسی هستند. برای
مشاهدۀ تفاوت آن با \lr{\texttt{private}}، می‌توانید در کلاس
\lr{\texttt{Animal}} تمامی فیلدها را \lr{\texttt{private}} کرده و نتیجه
را بررسی کنید.

رفتار متدهای \lr{\texttt{protected}} نیز مشابه فیلدها است؛ کلاس‌های فرزند
و کلاس‌های یک پکیج به آن‌ها دسترسی دارند. در مورد سازنده‌های
\lr{\texttt{protected}} نیز عملکردی کاملاً هم‌سان با متدهای
\lr{\texttt{protected}} وجود دارد.

پرسشی که مطرح می‌شود این است که تکلیف کلاس‌های \lr{\texttt{protected}}
چیست؟ وضعیتی مشابه کلاس‌های \lr{\texttt{private}} دارند. تنها زمانی
می‌توان کلاس \lr{\texttt{protected}} داشت که کلاس‌های تودرتو تعریف شده
باشند و صرفاً \lr{inner class} ها را بتوان \lr{\texttt{protected}} نمود.
این کلاس‌ها قابلیت ارث‌بری دارند و از طریق کلاس‌های درون همان پکیج یا
کلاس‌های فرزند می‌توان به آن‌ها دسترسی داشت.

\subsection{\lr{package-private}}

وقتی سطح دسترسی فیلد، متد یا کلاس را صریحاً مشخص نکنیم (به‌کار نبردن
\lr{\texttt{public}}، \lr{\texttt{private}} یا \lr{\texttt{protected}}
)، سطح دسترسی پیش‌فرض \lr{(default access)} یا همان \lr{package-private}
اعمال می‌شود. در این حالت، آن عضو تنها در درون همان پکیج قابل دسترسی است.

برای نمونه، یک پکیج به نام \lr{\texttt{zoo}} ایجاد کرده و به‌ترتیب
کلاس‌های \lr{\texttt{Zoo}} ، \lr{\texttt{Animal}} ، \lr{\texttt{Panda}} و
\lr{\texttt{Main}} را به‌صورت زیر تعریف کنید:

\begin{codebox}{java}
package zoo;
import java.util.ArrayList;
public class Zoo {
     String name;
     ArrayList<Animal> animals;

    public Zoo(){
    }
}

package zoo;
import java.util.ArrayList;
public class Animal {
     String name;
     int age;
     String species;

    public Animal(String name, int age, String species) {
        this.name = name;
        this.age = age;
        this.species = species;
    }
    @Override
    public String toString() {
        return "Name: " + name + ", Age: " + age + ", Species: " + species;
    }
}
\end{codebox}

با این کار، احتمالاً با خطا مواجه می‌شوید. دلیل آن است که کلاس
\lr{\texttt{Main}} درون پکیج \lr{\texttt{zoo}} قرار ندارد. چنانچه کلاس
\lr{\texttt{Main}} را نیز به این پکیج منتقل کنید و نتیجه را بررسی
نمایید، خواهید دید که خروجی مورد انتظار حاصل می‌شود. این دقیقاً مفهوم
\lr{package-private} است.

رفتار متدها و کلاس‌ها در سطح دسترسی \lr{package-private} نیز مشابه فیلدها
می‌باشد. درباره تفاوت میان \lr{package-private} و \lr{protected} میتوان
گفت این تفاوت در گستره دسترسی آن‌هاست: سطح دسترسی پیش‌فرض
\lr{(package-private)} عضوی را تنها برای کلاس‌های درون همان بسته قابل
مشاهده می‌سازد. اما \lr{protected} علاوه بر همان بسته، آن عضو را در تمام
کلاس‌های فرزند نیز (حتی اگر در بسته‌ای دیگر تعریف شده باشند) در دسترس قرار
می‌دهد. به‌بیان دیگر، \lr{protected} یک پل ارتباطی میان وراثت و بسته ایجاد
می‌کند، حال آنکه \lr{package-private} صرفاً محدود به مرز بسته است.

\section{چه چیزی یاد گرفتیم؟}

با مطالعه این مستند، با موارد زیر آشنا شدیم:

\begin{itemize}
\tightlist
\item
  مفهوم کپسوله‌سازی \lr{(Encapsulation)} و دلیل استفاده از آن.
\item
  تفاوت کلیدواژه‌های \lr{\texttt{private}} ، \lr{\texttt{protected}} ،
  \lr{\texttt{public}} و مفهوم \lr{package-private} .
\item
  چیستی و کاربرد متدهای \lr{getter} و \lr{setter} .
\end{itemize}
